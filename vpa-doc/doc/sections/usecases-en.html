Use cases: vehicle (type) authorisation and registration back to ToC

## Use case: Vehicle Authorisation Using the Verified Permissions, we can create for VA: ``` era:VehicleTypeAuthorisation rdfs:subClassOf vp:Permission era:VAApplication rdfs:subClassOf vp:Request era:VAApplicationCase rdfs:subClassOf vp:Case ``` A specific example is application V-20230310-NNN (number anonymised) as issued by erava:mf-MMMM (anonymised manufacturer). It concerned one Authorisation Case for a certain vehicle type and some vehicles for placing on the market in a certain Area of Use. This would then be stored as: ``` erava:V-20230310-NNN a era:VehicleTypeAuthorisation . erava:va-V-20230310-NNN a era:VAApplication . erava:vac-V-20230310-NNN-0 a era:VAApplicationCase . erava:mf-MMMM vp:requests erava:V-20230310-NNN ; vp:issues erava:va-V-20230310-NNN . erava:vac-V-20230310-NNN-0 vp:supports erava:va-V-20230310-NNN ; era:areaOfUse era:Network1 , era:Network2 , era:SubNetwork3 ; vp:permissionType era-skos-va:C2T . eratv:51-019-001-6-002-001 a era:VehicleType ; vp:definesCase erava:vac-V-20230310-NNN-0 . erava:vs-V-20230310-NNN-0 a erava:VehiclesCollection ; era:contains eravr:31717813003-2 , eravr:32717813002-4 , ... # and others eravr:31717813003-2 a era:Vehicle ; era:inCollection erava:vs-V-20230310-NNN-0 . ... # other vehicles in this authorisation case. ``` The properties from a `era:VehicleTypeAuthorisation` are mainly inherited from a vp:Permission: - The issue, starting and expiration dates - The status (excluding those which can be deduced from the dates) and the date the Permission was given that status. Example: 'REVOKED'. A detailed analysis of the Permissions' properties is ongoing. ## Use case: Vehicle Registration After these vehicles have been authorised in conformity to the given type and area of use, they can be registered by the Keeper for operation there. The instances created from that process could be: ``` eravr:V-20230310-NNN a era:VehicleRegistration . ```
